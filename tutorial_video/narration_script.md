5. 期刊要 code、reviewer 有質疑，你都拿得出來。
6. 核心優點二：速度——前期學一點，之後每篇論文分析時間大幅縮短。

## 段3 研究即工程專案 + 結構 + 資安 — CONCEPT + DEMO（~95s）
CONCEPT:
1. 把每一個研究，都當成一個工程專案來經營。
2. 工程專案的三個好習慣：結構化管理、版本控制、可驗證。
3. 固定結構：data（raw/processed）、figures、tables、manuscripts、notebooks，加 requirements.txt。
4. 鐵律：raw 裡的原始資料唯讀，永遠不要直接改。
5. 所有清理、轉換都用程式碼做，輸出到 processed。
6. 接著把專案放上 GitHub，建立一個 repository。
DEMO[vid5 @ mkdir]：終端機一行指令，建好專案資料夾。
DEMO[vid6 @ repo+subdirs]：建好子資料夾，並在 GitHub 開一個 repo。
CONCEPT（資安）:
7. 按按鈕前先記住：Public repo 全世界可見。
8. 就算設 Private，資料也已離開你的電腦，存在第三方伺服器。
9. Git 會留歷史：push 過的檔案，刪掉還是挖得出來。
10. 病人個資、API key、密碼，絕對不能 push。
11. 病人資料推上 GitHub＝違反 IRB，更是個資法資安事件。

## 段4 .gitignore 順序陷阱 — CONCEPT（~42s）
1. 防呆做法：一個叫 .gitignore 的隱藏檔。
2. 第一行寫上 data/，整個資料夾 Git 就當作不存在。
3. 陷阱：.gitignore 只對「還沒被追蹤」的檔案有效，不會回溯。
4. 先 commit 才補 .gitignore，data 已在追蹤清單，照樣會上傳。
5. 順序是關鍵：先寫好 .gitignore，再做第一次 commit。
6. 一句話：先關門，再搬東西。
7. 不小心追蹤了，可用 git rm -r --cached data/ 移除追蹤、保留本機檔案。

## 段5 add / commit / push — CONCEPT + DEMO（~70s）
CONCEPT（寄包裹比喻）:
1. Git 最核心三個動作：add、commit、push。最好的比喻是寄包裹。
2. git add＝把要打包的檔案裝進箱子。
3. git commit＝封箱，貼一張標籤；這就是一個可隨時回去的存檔點。
4. 標籤就是 commit message，例如「加入排除條件：BMI 缺失值」。
5. commit message 是寫給三個月後的自己看的。
6. git push＝把箱子寄到 GitHub。add、commit 都只在本機，push 才上傳。
7. 完整節奏：改檔案 → add → commit → push。
DEMO[vid6 @ git commands]：實際就是三行指令。
8. add . 會全選變動檔，但 data/ 已被 .gitignore 關在門外，病人資料永遠不會被裝箱。

## 段6 去識別化與兩條外洩路徑 — CONCEPT（~55s）
1. 規則：這個專案資料夾，從頭到尾不可以出現病人個資。
2. 資料外流有兩條路：第一條是 GitHub，我們用 .gitignore 關掉了。
3. 第二條是 AI——雲端服務讀到什麼，就會送到模型廠商的伺服器。
4. .gitignore 擋不住這條，因為 AI 讀檔不經過 Git。
5. 把可識別的病人資料交給雲端 AI＝把個資傳出院外，明確違法。
6. 做法：去識別化在院內完成，資料夾只放去識別化後的資料。
7. 姓名、病歷號、身分證字號 → 換成研究編號。
8. 出生年月日 → 換成年齡；日期 → 換成相對天數。
9. 對照表留在院內系統，永遠不放進專案資料夾。

## 段7 資料夾不是圍牆，機器才是 — CONCEPT（~52s）
1. 你可能想：個資放在資料夾外，AI 只在資料夾裡，應該安全了吧？
2. 很遺憾，這個假設不成立。
3. 更嚴格的規則：要跑雲端 AI 的機器，整台都不要有任何個資。
4. AI 的「工作資料夾」是預設範圍，不是一道圍牆。
5. 它寫入被限制在專案內，但讀取不是；它用的是你的權限。
6. 你以為 AI 只待在一個房間，其實它拿著整棟房子的鑰匙。
7. 它開別的房間會敲門問你，但用久了大家都會按同意按到麻痺。
8. 最可靠的不是「把門關好」，而是讓房子裡根本沒有貴重物品。
9. 這在資安叫「降低攻擊面」：個資和雲端 AI，永遠不要在同一台機器。

## 段9 EDA 與三個協作原則 — CONCEPT + DEMO（~135s）
CONCEPT:
1. 我的流程：Raw Data → EDA → Data Cleaning → 正式分析。
2. 順序很重要：先探索、再清理。沒看過資料，不知道該清什麼。
3. EDA 像體檢，先看出哪裡有問題；Cleaning 像治療，對症處理。
4. 三個協作原則。原則一：不要一次到底，一步一步來、看結果再走。
5. 原則二：每一步都要驗收——AI 交的東西，主治醫師要過目。
6. 原則三：臨床資料很髒，不要全自動清理；每個決策都是臨床判斷。
7. 第一個任務就是 EDA，來看我的 prompt 怎麼寫。
